# Reading Room Setup Guide

How to set up and configure the Reading Room for your project.

## What Is the Reading Room?

The Reading Room is your project's centralized working area for canonical documents. It holds specs, implementation plans, session checkpoints, branding assets, and research materials. Every repo in your project references this single location as the source of truth for project-level documentation.

The Library reads from the Reading Room for spec-based operations (audit, plan, compile) and writes to it for checkpoints and project state tracking.

## Two Modes

### Dedicated Repository (recommended)

A standalone git repo that contains only project-level documents. Best for multi-repo projects where multiple application repos need to reference the same specs and plans.

```
my-project/
  app-backend/          # git repo
  app-frontend/         # git repo
  library-reading-room/ # git repo  <-- Reading Room
  the-library/          # git repo
```

Advantages:
- Independent version history for specs and plans
- Clear ownership and access control
- No coupling between document changes and code changes
- Works naturally with multiple application repos

### Directory at Repo Root

A directory inside an existing repo. Suitable for single-repo projects or monorepos where co-locating documents with code is preferred.

```
my-project/
  src/
  tests/
  reading-room/   <-- Reading Room
  package.json
```

Advantages:
- Simpler setup for single-repo projects
- Documents travel with the code
- One fewer repo to manage

## Standard Structure

The Reading Room follows a fixed directory layout. These conventions are enforced by The Library and are not configurable:

```
reading-room/
  specs/           # Canonical spec files (the single source of truth)
  plans/           # Implementation plans
  checkpoints/     # Session checkpoint files
  branding/        # Brand kit, logos, colors, design tokens (optional)
  research/        # Research materials, competitor analysis (optional)
  standards/       # Coding and process standards (optional)
  CLAUDE.md        # Project-level instructions for Claude Code
  PROJECT-STATE.md # Current project state (managed by The Library)
```

### specs/

The most important directory. Contains the canonical spec files that define your project. These are the single source of truth -- all code, vault articles, and PM tasks trace back to specs.

### plans/

Implementation plans that break specs into actionable work. Typically generated by the `library:plan` skill or written manually.

### checkpoints/

Session checkpoint files written by `library:checkpoint`. Named with timestamp and topic:
```
2026-04-16-14-30-00-auth-middleware-refactor-checkpoint.md
```

### branding/

Optional. Centralized brand assets -- logos, color definitions, design tokens, brand kit HTML. Set up via `library:config branding`.

## Configuring library-config.yaml

The Library finds the Reading Room through the `reading_room` section in `library-config.yaml`:

```yaml
reading_room:
  path: /absolute/path/to/library-reading-room
  type: repo       # "repo" (dedicated repo) or "directory" (directory in a repo)
  branding: branding  # subdirectory name for brand assets (optional)
```

### For a dedicated repo:

```yaml
reading_room:
  path: /Users/me/projects/my-project/library-reading-room
  type: repo
```

### For a directory at repo root:

```yaml
reading_room:
  path: /Users/me/projects/my-project/reading-room
  type: directory
```

## Multi-Repo vs Monorepo Guidance

### Multi-repo projects

Use a **dedicated Reading Room repo**. Each application repo references the same Reading Room path in its `library-config.yaml`. This keeps specs versioned independently and avoids duplicating canonical documents across repos.

```yaml
# In compliance-core/library-config.yaml
reading_room:
  path: ../library-reading-room
  type: repo

# In compliance-ui/library-config.yaml
reading_room:
  path: ../library-reading-room
  type: repo
```

### Monorepo projects

Use a **directory at the repo root**. The Reading Room lives alongside your application code in the same repo.

```yaml
# In library-config.yaml at repo root
reading_room:
  path: ./reading-room
  type: directory
```

## Setup via library:config

The easiest way to set up the Reading Room is through the config skill:

```
/library:config
```

The first-run wizard asks:

> "Where is your Reading Room? (path to existing repo/directory, or 'create' to scaffold one)"

- If you provide an existing path, it validates the structure and checks for a `specs/` directory
- If you say "create", it scaffolds the standard directory layout at your chosen path

To update Reading Room settings later:

```
/library:config
```

Then select the Reading Room option from the update menu.
